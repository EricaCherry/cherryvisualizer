# Cherry — Research Appendix

Raw findings from the research workflow (9 web-grounded domains). This is the evidence behind the decisions in [PLAN.md](PLAN.md) and [ARCHITECTURE.md](ARCHITECTURE.md) — every key library/repo with its license and why it matters.

## 1. music-visualizer-engine-ecosystem

Six engines surveyed: MilkDrop (original, BSD, Windows-only plugin), Butterchurn (WebGL/JS port, MIT, npm-embeddable, best path for web apps), projectM (cross-platform C++ library, LGPL-2.1, native + WASM), Winamp AVS (BSD-3, modern port has libavs, niche use), GeISS (BSD-3, no preset system, not embeddable as library), MilkDrop3 (non-commercial freeware, not open-source usable). The fastest path to thousands of presets in a new app is: npm install butterchurn butterchurn-presets, then fetch supplemental packs from ansorre/tens-of-thousands-milkdrop-presets-for-butterchurn (15,056 presets, no stated license) or the Internet Archive 52k pack converted via jberg/milkdrop-preset-converter-node. The entire Milkdrop preset corpus (52k-130k+ .milk files) is treated as effectively public domain by the community, with no known enforcement of individual author copyrights.

**Techniques**
- Milkdrop .milk format: plain-text INI-like key=value file. Sections: [preset init code], [preset per frame N], [preset per pixel N], [per frame N], [per vertex N], custom shapes/waves, warp shader (HLSL), composite shader (HLSL). Variables: bass, mid, treb, vol (instantaneous), bass_att, mid_att, treb_att, vol_att (smoothed/attenuated). Q-variables q1-q32 bridge between code sections. Grid-based 'per-pixel' (actually per-mesh-vertex, not true per-pixel pre-shader era). MilkDrop 2 added programmable HLSL warp and composite pixel shaders.
- Butterchurn JSON format: .milk files are parsed and transpiled to JSON objects containing equation strings and shader GLSL code. The milkdrop-preset-converter tool handles this transpilation. JSON presets contain fields for per-frame equations (as strings), per-vertex equations, warp shader GLSL, composite shader GLSL, and parameter floats. Loaded via visualizer.loadPreset(presetJson, blendTime).
- Audio pipeline (Butterchurn): Web Audio API source node -> AnalyserNode (internal) -> FFT -> bass/mid/treb band detection -> equation variables. Input to Butterchurn is any Web Audio API AudioNode; the library creates its own internal AnalyserNode.
- Audio pipeline (projectM): Host pushes raw PCM float samples via addPCMfloat(float* samples, int count) or addPCMfloat_2ch() for stereo. Library handles its own FFT and beat detection internally. Typical call: push 512 samples per render frame.
- Preset blend transitions: Both Butterchurn and projectM support smooth crossfade transitions between presets. Butterchurn: loadPreset(preset, blendTime) where blendTime is seconds. This is implemented as a shader-level mix between outgoing and incoming preset render buffers.
- AVS preset format: Legacy binary .avs (component tree encoded as binary blob). Modern grandchild/vis_avs also writes/reads JSON. Components include: Super Scope (oscilloscope equations), Simple scope, Movement (per-pixel warp), Dynamic Movement (equation-based warp), Brightness, Blur, Channel Shift, and ~60 other effect components. No pixel shaders; component-based pipeline.
- GeISS rendering: No preset file format. Entirely procedural C++ with fixed visualization algorithms. Reacts to audio via DirectSound buffer analysis. Cannot load external presets. Not relevant to preset ecosystem.

**Recommendations**
- For a web app: use npm install butterchurn butterchurn-presets. This is a 2-package install that gives you a working WebGL2 Milkdrop renderer plus hundreds of bundled presets in under 10 minutes. Requires WebGL2-capable browser (all modern browsers qualify).
- To reach thousands of presets in a web app without conversion work: clone ansorre/tens-of-thousands-milkdrop-presets-for-butterchurn alongside butterchurn-presets. Combined total is 15,000+ presets already in Butterchurn JSON format, zero conversion needed.
- To access the full 52k-130k corpus: download the Internet Archive Milkdrop 2 52k pack and run jberg/milkdrop-preset-converter-node in batch mode. This converts .milk files to Butterchurn JSON. Both tools are MIT-licensed.
- For a native (non-web) app: use libprojectM (LGPL-2.1). Build and link as a shared library to satisfy LGPL requirements. Pair with the 9,795-preset cream-of-the-crop pack for the best out-of-box experience. Use the projectm-eval expression library (MIT) if you need the equation parser in isolation.
- For mobile or cross-platform without a GL context: projectM has Android support and Emscripten/WASM builds. The Emscripten example repo demonstrates browser embedding of the native C++ engine as an alternative to Butterchurn if GLSL shader compatibility is a concern.
- Preset quality filtering: the cream-of-the-crop pack (9,795 presets) is human-curated. The 52k+ packs include significant quantity of low-quality or broken presets. For a production app, start with cream-of-the-crop or butterchurn-presets curated bundles, then add the larger packs as optional user-loadable content.
- Avoid AVS for new apps unless you specifically target the AVS aesthetic. The preset ecosystem is orders of magnitude smaller than Milkdrop, the format is incompatible with Milkdrop, and the grandchild/vis_avs port is Windows-primary and not yet stable for library embedding.

**Pitfalls**
- LICENSE TRAP — MilkDrop3 (milkdrop2077/MilkDrop3): Non-commercial freeware only. Commercial use requires paid PRO license. Do not embed or redistribute for a commercial product without paying.
- LICENSE TRAP — milkdrop2077 preset generator tool: GPL-3.0. If you incorporate its code into your app, your entire app must be GPL-3. The tool is for generating/mashing presets offline, not for runtime embedding.
- LICENSE TRAP — projectM static linking: LGPL-2.1 requires that if you statically link libprojectM, end users must be able to relink the library. Build as a shared library (.dll/.so/.dylib) to avoid this obligation.
- PRESET LICENSE AMBIGUITY: No Milkdrop preset pack has a clean permissive license (CC0/MIT/Apache). The community assumption of public domain is informal. For a high-stakes commercial product, get legal review before mass-distributing preset packs. The projectM org's LICENSE.md acknowledges this explicitly.
- ansorre/tens-of-thousands-milkdrop-presets-for-butterchurn has NO stated license. The repo author states they are still converting more presets. Do not treat as definitively cleared for all commercial uses.
- Butterchurn requires WebGL2. Older browsers and some embedded WebViews do not support WebGL2. Always call butterchurn.isSupported() and provide a fallback.
- Butterchurn package version split: stable is 2.6.7 (npm tag 'latest'); 3.0.0-beta.5 is experimental. The beta uses WebAssembly for equation evaluation (faster) but may have preset compatibility regressions. Pin to 2.6.7 for production.
- GeISS has no preset ecosystem — it is a single procedural visualizer. Do not include it in any plan involving 'thousands of presets'. Its only value is BSD-3 reference code for the waveform-to-image math that predates Milkdrop.
- The Internet Archive 52k pack and 130k MegaPack are not officially organized or licensed. They contain presets by hundreds of individual authors over 20 years. Some presets reference textures that are not included and will render as blank or corrupted. The projectM texture pack repo (presets-milkdrop-texture-pack) is needed alongside them.
- AVS presets are a completely separate format and ecosystem from Milkdrop .milk presets. They are not interchangeable. A Butterchurn or projectM engine cannot load .avs files without a separate AVS renderer.

## 2. GLSL/Shader ecosystem for audio-reactive visuals

The audio-reactive GLSL shader ecosystem spans Shadertoy (dominant community hub, NC-locked by default), ISF (open spec, BSD-licensed tooling), glslCanvas/glslViewer (MIT/BSD standalone runners), three.js ShaderMaterial (MIT, most flexible for web), and several live-coding tools. The critical legal trap: Shadertoy's default license is CC-BY-NC-SA 3.0, which blocks all commercial use and ShareAlike infects derivatives. Permissive shader collections exist but are scattered. Audio data is universally fed as either (a) a 1D/2D texture uploaded from AnalyserNode Uint8Arrays via texImage2D/texSubImage2D each frame, or (b) pre-computed scalar uniforms (bass, treble, beat). Shadertoy's canonical format is a 512x2 Uint8 texture: row 0 = FFT spectrum (dB-scaled, 0–255), row 1 = waveform (centered at 128). ISF uses float textures of variable width per channel.

**Techniques**
- Shadertoy iChannel audio texture format: 512x2 Uint8 texture (GL_RED/GL_UNSIGNED_BYTE). Row 0 = FFT spectrum: 2048-sample FFT → 1024 bins → 512 pixels, Blackman-windowed, magnitude in dB clamped [-100,-30] and scaled to 0–255. Row 1 = waveform: mono mix L+R, normalized -16384..16384 → 0..255, centered at 128. Sample in GLSL: float fft = texture(iChannel0, vec2(freq, 0.25)).r; float wave = texture(iChannel0, vec2(t, 0.75)).r;
- Web Audio API → WebGL texture upload pattern: (1) Create AudioContext and AnalyserNode (fftSize=1024 for 512 bins). (2) Allocate Uint8Array of length analyser.frequencyBinCount. (3) Each frame: analyser.getByteFrequencyData(freqData) and analyser.getByteTimeDomainData(waveData). (4) Pack into combined 512x2 Uint8Array or upload separately. (5) gl.texImage2D(gl.TEXTURE_2D, 0, gl.RGBA, length, 1, 0, gl.RGBA, gl.UNSIGNED_BYTE, rgbaBuffer) — replicate each value across R,G,B channels with A=255. Use gl.texSubImage2D for per-frame updates to avoid reallocating. Set TEXTURE_MIN_FILTER=LINEAR, WRAP_S/WRAP_T=CLAMP_TO_EDGE.
- Three.js ShaderMaterial audio texture pattern: const fftData = new Uint8Array(analyser.frequencyBinCount); const fftTex = new THREE.DataTexture(fftData, analyser.frequencyBinCount, 1, THREE.RedFormat, THREE.UnsignedByteType); In render loop: analyser.getByteFrequencyData(fftData); fftTex.needsUpdate = true; Uniform: {u_fft: {value: fftTex}}. GLSL: uniform sampler2D u_fft; float amp = texture2D(u_fft, vec2(uv.x, 0.5)).r / 255.0;
- ISF audio input declaration in JSON metadata: {"INPUTS": [{"NAME": "audioData", "TYPE": "audio", "MAX": 512}, {"NAME": "fftData", "TYPE": "audioFFT", "MAX": 512}]}. GLSL access: vec4 sample = IMG_NORM_PIXEL(fftData, vec2(freqNorm, channelNorm)); Waveform amplitude centered at 0.5. FFT amplitude is linear magnitude. Preferably 32-bit float textures.
- Bonzomatic audio uniforms: uniform sampler1D texFFT; — raw FFT per frame. uniform sampler1D texFFTSmoothed; — temporal smoothing applied. uniform sampler1D texFFTIntegrated; — accumulates over time (grows monotonically). Sample: float bass = texture(texFFT, 0.05).r; float treble = texture(texFFT, 0.8).r;
- Beat detection uniform patterns (implementation-dependent): Binary float (0.0 or 1.0 for one frame per beat) — Audio-Shader-Studio uses u_beatDetected. Incremental counter — u_beatCount (Audio-Shader-Studio), syn_BeatTime (Synesthesia). BPM-synced oscillators — syn_BPMSin, syn_BPMTri (Synesthesia). Onset/transient detection via spectral flux — u_onsetDetected (Audio-Shader-Studio), syn_Hits (Synesthesia). In custom WebGL: compute RMS, low-pass filter it, threshold crossing = beat.
- Shadertoy → standalone: Uniform mapping. iTime → elapsed seconds (float). iResolution → vec3(width, height, 1.0). iMouse → vec4(x, y, clickX, clickY). iFrame → int. iChannelN → sampler2D (or sampler3D for volume). iChannelResolution[N] → vec3. iDate → vec4(year, month, day, seconds). iSampleRate → float (44100.0). TouchDesigner mapping: iTime → absTime.seconds, iChannelN → sTD2DInputs[N].
- Scalar audio uniforms (pre-computed, no texture): Pass float uniforms for bass/mid/treble energy from Web Audio API by averaging frequency bins. Example: bassLevel = mean(freqData[0..10]) / 255.0; midLevel = mean(freqData[10..100]) / 255.0; trebleLevel = mean(freqData[100..255]) / 255.0. Update with material.uniforms.u_bass.value = bassLevel each frame. This is simpler than texture upload and sufficient for reactive displacement or color changes.
- glslify module include pattern for shader utilities: #pragma glslify: noise = require('glsl-noise/simplex/3d'); #pragma glslify: map = require('glsl-map'); Processed by glslify CLI or webpack/rollup plugin. Enables reuse of npm-published MIT-licensed GLSL functions (glsl-noise, glsl-easings, glsl-hsv2rgb etc.) without copy-pasting.

**Recommendations**
- Do NOT use Shadertoy shaders without explicit per-shader license verification. The platform default CC-BY-NC-SA 3.0 prohibits commercial use AND requires ShareAlike on derivatives. A GLSL port counts as a derivative work legally. Check the shader's code comments or description for an explicit MIT/CC-BY/public domain declaration before adapting.
- For legally safe permissive shader sources, prefer: (1) Shadertoy shaders by iq (Inigo Quilez) that declare MIT in their code comments — he explicitly MIT-licenses many educational shaders. (2) Vidvox ISF-Files collection (MIT, 200+ shaders). (3) gl-transitions (MIT). (4) Any shader with explicit MIT, CC0, CC-BY (without NC or SA), BSD, or Unlicense declaration.
- Use Audio-Shader-Studio (sandner-art, MIT) as a reference implementation for audio uniform naming: it provides the most complete set of audio-reactive uniforms (u_frequencyTexture, u_timeDomainTexture, u_bassLevel, u_trebleLevel, u_beatDetected, u_beatCount, u_onsetDetected) and is legally safe to study and adapt.
- For a Shadertoy-compatible web runner, use glslCanvas (MIT) or glsl-canvas-js (MIT) and manually bind an audio texture to iChannel0. These do not auto-provide audio — you inject it via setUniform() with a WebGL texture handle each frame.
- For desktop standalone Shadertoy running WITH audio support, use Shadertron — it explicitly supports Microphone and Audio input sources mapping to iChannel. Verify its license before commercial use (not confirmed in research).
- For live-coding/performance use, Bonzomatic (Unlicense/public domain) provides the most liberal license. Its texFFT/texFFTSmoothed/texFFTIntegrated sampler1D uniforms are well-designed for real-time performance.
- LYGIA is a licensing trap: despite appearing in many shader tutorials as a utility library, it is under the Prosperity License (NOT MIT/BSD). Do not use it in commercial products without paying for a patron license. For MIT-licensed GLSL utilities, use individual glslify npm packages (glsl-noise, glsl-easings, etc.) instead.
- For the ISF ecosystem, all tooling (VVISF-GL, ISFGLSLGenerator, ISFMSLKit) is BSD licensed. The ISF spec itself (mrRay/ISF_Spec) is MIT. This is the safest ecosystem for building audio-reactive shader tools commercially.
- When implementing FFT texture upload, use gl.texSubImage2D (not texImage2D) for per-frame updates — it avoids GPU memory reallocation. Set fftSize to 2048 on AnalyserNode to get 1024 bins; take only the first 512 to match Shadertoy's layout. Use CLAMP_TO_EDGE wrapping and LINEAR filtering.
- For beat detection without a dedicated library: compute short-term energy (sum of squared amplitudes in bass band, ~20-250 Hz bins), compare to a 43-frame (~1 second) history average, declare a beat when current energy exceeds 1.3-1.5x the historical average. Pass as a float uniform that stays 1.0 for one frame then returns to 0.0.

**Pitfalls**
- LICENSE TRAP — Shadertoy CC-BY-NC-SA 3.0 default: The NC clause prohibits ANY commercial use. ShareAlike requires derivatives to use the same license. A GLSL port to another language or platform is legally a derivative work. Most Shadertoy shaders (including famous ones by artists other than iq) carry this restriction by default. Assume CC-BY-NC-SA unless explicit MIT/CC0 is declared IN the shader code.
- LICENSE TRAP — LYGIA Prosperity License: Widely referenced in tutorials as if it were MIT/BSD (it historically was BSD, but the license changed). Current LYGIA is Prosperity Public License 3.0, which explicitly prohibits commercial use beyond a 30-day trial. Do not bundle LYGIA into any commercial product without licensing.
- LICENSE TRAP — The Book of Shaders (patriciogonzalezvivo): All rights reserved — NOT MIT or permissive. Cannot legally copy examples for commercial use.
- LICENSE TRAP — iquilezles.org articles: No site-wide permissive license. Individual articles assert copyright. The author's Shadertoy shaders may be MIT-licensed (check code comments), but the iquilezles.org article code snippets have no confirmed permissive license.
- LICENSE TRAP — shader-web-background (xemantic): GPL v3. Using it in a web app makes your app subject to GPL copyleft. Not suitable for closed-source commercial projects.
- TECHNICAL TRAP — Shadertoy FFT uses 44.1kHz not 48kHz: Some documentation references 48kHz, but browsers typically use 44.1kHz. The 512 pixels represent only the lower half of the spectrum (0–11,025 Hz), not the full range. The upper half is discarded.
- TECHNICAL TRAP — Shadertoy FFT is dB-scaled, not linear: The spectrum values in row 0 are NOT linear magnitude. They are: dB = 20*log10(magnitude), clamped to [-100, -30] dB, then scaled to 0–255. A value of 0 means -100 dBFS, not silence. Do not compare Shadertoy FFT values directly to linear FFT outputs from Web Audio API's getFloatFrequencyData().
- TECHNICAL TRAP — texImage2D vs texSubImage2D: Using texImage2D every frame causes GPU memory reallocation. Always use texSubImage2D for per-frame audio texture updates after initial creation.
- TECHNICAL TRAP — glslCanvas and glsl-canvas-js provide NO audio texture support out of the box. Many tutorials imply they are Shadertoy-compatible — they are not for audio. You must implement the Web Audio API integration yourself and inject the texture.
- ECOSYSTEM TRAP — glslify is functionally unmaintained (last release 2016). The stack.gl ecosystem it belongs to has largely stalled. Individual npm glsl-* packages may still work but receive no security updates. Use only for build-time shader preprocessing, not runtime.
- COMPATIBILITY TRAP — ISF standard audio functions (IMG_NORM_PIXEL) are non-standard GLSL macros injected by ISF hosts. Shaders using these will not run directly in WebGL without a preprocessor that replaces them with standard texture2D/texture calls.
- Shadertoy API requires 'Public + API' privacy setting on shaders: many shaders are public but not API-accessible. The API endpoint is https://www.shadertoy.com/api/v1/shaders/{id}?key={appkey} and requires a free API key. Even with API access, the shader's CC-BY-NC-SA license still applies to the code returned.

**Techniques**
- PLASMA: Superimpose 3–5 sine waves over normalized UV coordinates (sin(x+t), sin(y+t*0.7), sin(sqrt(x^2+y^2)+t), etc.), sum them, feed into a cosine color palette (iquilez technique: color = a + b*cos(2pi*(c*val+d)) per channel). GPU cost: trivial. Audio binding: modulate per-wave frequency multipliers with FFT band magnitudes; high bass → faster, larger waves. Color palette phase driven by spectral centroid.
- TUNNEL: Convert fragment UV to polar coordinates (r = 1/length(uv), a = atan(uv.y, uv.x)). Use (r + time) as texture V and (a/PI) as texture U, or drive a procedural checkerboard. Fog = exponential darkening by 1/r. Audio binding: speed of time offset proportional to BPM; camera wobble (UV offset) tied to mid frequencies; tunnel radius pulsing with bass.
- ROTOZOOMER: In fragment shader compute rotated+scaled UV via 2x2 rotation matrix (cos/sin of time) applied to centered UV, then sample a tiled texture or procedural pattern. Scale factor oscillates. Audio binding: rotation speed = bass, zoom amplitude = overall RMS, tile pattern phase = beat clock.
- METABALLS/BLOBS: Two approaches — (1) 2D: compute sum of falloff functions (k/dist^2) for each blob center; threshold at 1.0; color by gradient of field. (2) 3D GPU marching cubes (sjpt/metaballsWebgl): 4-pass GPU pipeline — spatial subdivision, potential grid fill, voxel relevance, triangle generation. Audio binding: blob radii/speeds scale with bass amplitude; number of active blobs follows beat count.
- COPPER BARS: Per-fragment: bar_center[i] = sin(time * freq[i] + phase[i]) * 0.4 + 0.5; if abs(uv.y - bar_center) < bar_width, color = gradient(uv.y - bar_center). Palette cycling: rotate hue over time. Audio binding: bar count = frequency bin count; bar vertical positions driven by FFT magnitudes across 8–16 bands; width tracks RMS.
- STARFIELD: Two variants — (1) Particle: per-star z decreases each frame, perspective-project (x/z, y/z) to screen, brightness = (1-z). (2) GLSL procedural: hash(floor(uv * scale)) gives star positions per cell; soft disk with glow. Audio binding: flight speed proportional to bass; star brightness modulated by high-frequency energy; spawn density follows beat.
- FIRE (DOOM-style): Maintain two framebuffer textures. Each frame: read pixel at (x, y+1) and neighbors, average with darkening coefficient < 1, write to alternate buffer. Inject heat (255) at bottom row. GPU implementation in skeeto/webgl-fire does all propagation in fragment shader via texture ping-pong. Audio binding: heat injection amplitude scales with bass; spread variance with mid frequencies; overall brightness with RMS.
- RAYMARCHED SCENES: March ray from camera origin along direction vector; step by distance returned by scene SDF each iteration; stop when distance < epsilon or step count exceeded (sphere tracing). Scene SDFs: sdSphere, sdBox, sdTorus (from glsl-sdf-primitives). Lighting: analytical normal from gradient (finite difference of SDF), diffuse + specular + ambient occlusion (cone sampling). Audio binding: camera orbit speed with bass, SDF smooth-blend factor with treble, domain repetition scale with RMS.
- KALEIDO-IFS: Kaleidoscope — convert UV to polar, fold angle with modulo(a, 2*PI/N)*N (N = symmetry count), then reflect; sample pattern in folded space. IFS — iteratively apply affine transforms (scale + rotate + translate) with probability weights; accumulate point histogram. Audio binding for kaleido: N-fold symmetry changes per beat; rotation speed with BPM. For IFS: transform weights driven by FFT band energies.
- VOXEL LANDSCAPE: Back-to-front column casting: for each screen column, cast ray from camera; sample color/height map at (x + t*dx, y + t*dy) for increasing t; perspective-project height difference to screen Y; draw vertical line segment in color from map. Can substitute Perlin noise heightmap for Comanche assets. Audio binding: fly speed (dt per frame) proportional to bass; heightmap vertical scale follows RMS; fog density modulated by treble.
- TWISTER: Slice a vertical column into N horizontal bands. Each band i has its UV X-coordinate offset by sin(time * speed + i * twist_freq) * amplitude, creating a lateral sine wave per slice. Optionally apply per-band hue rotation. Audio binding: twist amplitude scales with bass; twist_freq with spectral centroid; number of visible bands with RMS.
- BUMP MAPPING: Procedural bump: define height field h(uv) = noise(uv * scale + time). Derive surface normal analytically or via dFdx/dFdy screen-space derivatives: n = normalize(vec3(-dFdx(h), -dFdy(h), 1.0)). Apply diffuse/specular lighting with this normal. Audio binding: noise scale with spectral centroid; bump height amplitude with bass; light position orbits driven by BPM.
- FEEDBACK/ZOOMER: Render to FBO A with scene content. In next frame: sample FBO A with slightly zoomed/rotated UV (e.g., uv = uv * 0.99 + 0.005 + rotation), multiply by decay < 1.0, write to FBO B. Composite live scene over decayed feedback. Implement via ping-pong framebuffers (pixlpa/WebGL-Feedback-Demo pattern). Audio binding: zoom factor driven by bass; rotation speed by BPM; decay (persistence) by treble; live scene intensity by RMS.
- LISSAJOUS: XY oscilloscope mode — collect N audio samples per frame (left → X, right → Y). Render as GL_LINE_STRIP or additive point cloud via WebGL2. Add persistence by rendering to FBO and blending with decay. For pure GLSL: parametric curve in fragment shader: pos(t) = vec2(sin(w_x*t + phi), sin(w_y*t)). Audio binding: w_x and w_y derived from dominant frequency peaks in L/R channels; trail decay from RMS.
- MOIRE: Per-fragment compute distances d1 and d2 from two focal points. Apply sin(freq*(d1 - d2)) and cos(freq*(d1 + d2)) to RGB channels (Apache-2.0 algorithm from leapjs gist). Interference naturally emerges. Audio binding: focal point separation driven by bass amplitude; frequency multipliers (425, 720, 555 in reference) modulated by FFT bands; focal point positions animated at BPM speed.

**Recommendations**
- Start from sandner-art/Audio-Shader-Studio (MIT) for the audio-binding layer: it already exposes u_bass, u_treble, u_rms, u_beatCount, u_spectralCentroid, plus full FFT and waveform textures as samplers. Write each effect as a standalone fragment shader dropped into this framework.
- Use ashima/webgl-noise (MIT) as the noise primitive for plasma, bump mapping, fire spread noise, and voxel heightmap generation — it is self-contained GLSL with no external data files and covers simplex, classic Perlin, cellular, and psrdnoise.
- Use glslify (MIT) + glsl-sdf-primitives (MIT) + glsl-sdf-ops (MIT) to compose raymarched scenes modularly. This keeps each SDF primitive in a separate npm-installable module and avoids monolithic shader files.
- For the feedback/zoomer effect, implement a minimal two-FBO ping-pong directly in raw WebGL rather than pulling in a framework: bind FBO A for writing, sample FBO B (previous frame) with scaled UV, write decayed result, then swap. This avoids any dependency.
- For Lissajous, use oscilloscope.js (MIT) XY mode as the audio→coordinate mapping reference, but render via WebGL2 additive blending with a custom fragment shader for glow/persistence instead of its Canvas2D output.
- For the voxel landscape, generate the heightmap from ashima simplex noise rather than using VoxelSpace's Comanche-derived assets. The MIT code logic is clean; only the bundled PNG assets carry IP risk.
- Use mrkite/demofx (BSD-2) as algorithm pseudocode reference for tunnel, rotozoomer, metablobs, and moire when porting to GLSL — the TypeScript is readable and the BSD-2 license allows direct code reuse with attribution.
- Do NOT use LYGIA shader library in any project that may become commercial. It is Prosperity Public License (non-commercial; 30-day trial only). Equivalent functionality exists in the MIT-licensed glslify ecosystem.

- For butterchurn (MIT): it bundles hundreds of Milkdrop presets covering plasma warp, tunnel, feedback, and kaleido patterns. Use it as a high-fidelity reference for how Milkdrop's per-pixel and per-vertex equation system maps to audio uniforms.

## 4. Web game frameworks and physics engines for audio-driven game-based visualizer modes

As of June 2026: Phaser 4 (released April 10, 2026) is the most mature full-featured 2D game framework for audio-driven games — MIT licensed, with built-in WebAudioSoundManager exposing the AudioContext directly, a physics-integration plugin (phaser-box2d, MIT, 65KB gz), and a new RenderNode WebGL2 renderer capable of 1M sprites in a single draw call via SpriteGPULayer. PixiJS v8 (latest 8.18.1, MIT) is the best pure-renderer choice: exposes an AnalyserNode via @pixijs/sound's audio chain, fastest raw WebGL/WebGPU throughput, and is what Bemuse (rhythm game) and webosu (osu clone) use in production. KAPLAY (MIT, v3001) and melonJS (MIT, v19.6) are viable lighter alternatives. For physics: Rapier2d (Apache-2.0) is the only engine with an explicit cross-platform deterministic NPM build (@dimforge/rapier2d-deterministic), critical for video export. matter.js (MIT, v0.20) added a fixed-timestep runner but determinism is not guaranteed across platforms. planck.js (MIT, v1.5) is a Box2D JS port with good determinism by design. p2-es (fork of abandoned p2.js) is MIT and maintained by pmndrs. box2d-wasm (zlib, ~162KB) is a direct Box2D v2.4 WASM port. No purpose-built 'audio-driven game engine' exists; the pattern is WebAudio AnalyserNode → getByteFrequencyData/getFloatTimeDomainData → mutate physics body velocities each game tick. Bemuse (AGPLv3, TRAP) and webosu (MIT code, CC-BY-NC media) are the only notable open-source web rhythm games using PixiJS. For video export, the correct approach is canvas.captureStream(0) + track.requestFrame() per rendered frame, combined with a deterministic physics stepper; MediaRecorder real-time capture is unreliable for frame-perfect export.

**Techniques**
- WebAudio AnalyserNode pattern: attach AnalyserNode to the game audio context, call getByteFrequencyData(uint8Array) or getFloatTimeDomainData(float32Array) once per game tick, map frequency bin amplitudes to physics body impulses, velocities, or spawn rates.
- Audio-driven Breakout: each game tick, read waveform amplitude at ball's x-position frequency bin, add scaled impulse to ball's y-velocity via physics engine. Phaser Arcade Physics or planck.js body.applyLinearImpulse() are simplest integration points.
- Audio-driven runner: BPM-sync platform spawn timing using AudioContext.currentTime + beat interval math; spawn platforms at beat boundaries, speed game scroll to tempo.
- Deterministic video export: use canvas.captureStream(0) to get a MediaStream with manual frame control, call track.requestFrame() after each rendered frame, drive physics with fixed-delta stepping (Rapier2d-deterministic or planck.js), advance AudioContext time synthetically via OfflineAudioContext pre-render.
- OfflineAudioContext pre-render: render entire audio track offline first to extract per-frame frequency arrays, then feed these pre-computed arrays into game loop during video export — eliminates real-time audio non-determinism entirely.
- PixiJS + @pixijs/sound audio chain: the analyser node sits at the head of the chain (analyser > compressor > destination), so spectrum data reflects pre-compression signal. Use sound.analyser directly for raw data.
- Phaser 4 SpriteGPULayer for particle visualizers: store audio-reactive particle positions in a static GPU buffer, update only changed entries, render 100K+ sprites in a single draw call — suitable for waveform particle backgrounds behind gameplay.
- Fixed-delta physics stepping pattern for video export: decouple render loop from physics loop, step physics engine N times per exported frame using fixed delta (e.g. 1000/60 ms), accumulate remainder for next frame.
- Rapier WASM determinism caveat: all simulation init values must themselves be deterministic — avoid Math.sin, Math.cos, Date.now(), Math.random() during body initialization; pre-compute with a seeded PRNG or from pre-rendered audio data.
- KAPLAY fixedDt/fixedUpdate: use the recently added fixedUpdate callback for physics-rate logic separate from render rate, matching the pattern needed for audio-driven state updates.

**Recommendations**
- For a Breakout-style visualizer with audio-driven ball physics: use Phaser 4 (MIT) + phaser-box2d (MIT) + Phaser's built-in WebAudioSoundManager. Wire AnalyserNode data to box2d body impulses in the update loop. This stack is entirely MIT, battle-tested, and has the most documentation for audio+physics integration.
- For highest rendering performance (particle-heavy waveform backgrounds): use PixiJS v8 (MIT) as the renderer + @pixijs/sound (MIT) for audio + planck.js (MIT) or rapier2d (Apache-2.0) for physics. PixiJS benchmarks at ~47 FPS on 10K sprites vs Phaser's 43 FPS; for 100K+ use PixiJS with GPU batching.
- For deterministic video export: mandate @dimforge/rapier2d-deterministic (Apache-2.0) as the physics engine. Pre-render audio via OfflineAudioContext to extract per-frame frequency arrays. Step Rapier with fixed delta. Use canvas.captureStream(0) + track.requestFrame() per frame. Do not use matter.js (cross-platform determinism not guaranteed).
- Avoid KAPLAY for performance-sensitive visualizers: benchmarks show ~3 FPS at 10K sprites. Use it only for simple, low-entity games where its beginner-friendly API is the priority.
- Do NOT use Bemuse as a code dependency: it is AGPLv3 which requires your entire web app to be open-sourced. Study it as an architecture reference only.
- Flag the CC-BY-NC trap in webosu: the media assets are non-commercial only. If building a commercial product, any webosu-derived asset pipeline must strip all ppy-copyrighted content.
- For a runner mode (BPM-synced obstacles): melonJS (MIT, v19.6) is a strong lightweight choice — it has getAudioContext() and getMasterGain() for custom WebAudio graphs, GPU-accelerated tilemap rendering, and is under 100KB gz. Combine with p2-es (MIT) for spring-based runner physics.
- planck.js (MIT) is preferred over box2d-wasm (Zlib) when WASM is undesirable (e.g. CSP constraints, simpler build pipeline). For maximum 2D physics performance, rapier2d-simd (Apache-2.0) is 2-5x faster than any pure-JS engine as of 2025.
- Phaser 4 is the safest all-in-one choice for a production audio visualizer game: MIT, WebGL2, built-in audio context access, Box2D v3 plugin (MIT), SpriteGPULayer for particle effects, and the largest community for support.
- For the audio feature stream integration pattern: pre-compute a Float32Array[] of per-frame frequency data using OfflineAudioContext before any game or export loop starts. This makes the audio pipeline fully deterministic and decouples it from browser audio scheduling jitter.

**Pitfalls**
- Bemuse is AGPLv3: any web app that modifies and serves Bemuse code must publish its full source. This is a viral copyleft license — do not use as a library dependency.
- webosu media assets are CC-BY-NC: even though the code is MIT, ppy beatmap and audio assets are non-commercial. Shipping these in a product is a license violation.
- matter.js cross-platform determinism is NOT guaranteed despite fixed-timestep runner: GitHub issue #1190 documents divergence between Node.js and browser execution. Do not use for video export where bit-identical physics replay is required.
- KAPLAY rendering performance is approximately 3 FPS at 10K sprites (Shirajuki benchmark): completely unsuitable for particle-heavy audio visualizers. Its strength is simplicity, not throughput.
- rapier2d Apache-2.0 is permissive but not MIT: Apache-2.0 has patent retaliation clauses. If your project has strict MIT-only dependency policies, this may require legal review. It is NOT GPL and IS commercially usable.
- Math.sin/Math.cos in Rapier init breaks cross-platform determinism: Rapier's determinism guarantee explicitly carves out transcendental functions as non-deterministic across platforms. Any use of these in body initialization must be replaced with pre-computed lookup tables.
- OffscreenCanvas breaks MediaRecorder capture pipeline: some deterministic video rendering approaches disable OffscreenCanvas (by overriding window.OffscreenCanvas = undefined) because it bypasses the main-thread canvas capture pipeline.
- Phaser 3 physics (Arcade + Matter) vs Phaser 4: Phaser 4 dropped the built-in Matter.js integration. The new physics solution is phaser-box2d (separate MIT package). Existing Phaser 3 + Matter.js codebases must migrate.
- p2.js (original schteppe repo) is abandoned — last published 10 years ago on npm. Only use p2-es (pmndrs fork) which is actively maintained. The original p2 package on npm will receive no security patches.
- box2d-wasm (Birch-san) targets Box2D v2.4, not v3: Box2D v3 (Erin Catto) has a different API and is what phaser-box2d uses. The two WASM builds are not API-compatible — verify which Box2D version your physics code targets.
- WebGPU backend in PixiJS v8 does not always beat WebGL: PixiJS documentation notes WebGPU may underperform WebGL when CPU-side batch breaks are infrequent. Do not assume WebGPU = faster without profiling your specific scene.
- Phaser 4 SpriteGPULayer memory cost: 168 bytes per sprite on both CPU and GPU. At 1M sprites this is ~168MB GPU VRAM — verify target hardware can accommodate this before committing to the architecture.

## 5. Music Visualizer DSP Techniques

Traditional music visualizers decompose audio into frequency, time, or phase-domain representations using the FFT (or CQT/SWIFT variants) as the core DSP primitive. The key perceptual insight driving every technique is that human hearing is logarithmic in both frequency and amplitude: a log-frequency axis and dB magnitude scale must be used to produce displays that feel natural. Seven canonical visualizer types each expose a different facet of the signal: spectrum bars (energy vs. frequency snapshot), oscilloscope/waveform (time domain), Lissajous/vectorscope (inter-channel phase), spectrogram/waterfall (time-frequency energy history), radial/polar spectrum (same as bars but mapped to polar coordinates), chromagram (pitch-class energy folded across octaves), and constant-Q (true log-frequency bins matching musical scale). Best practices are: apply a window function (Hann at 50% overlap, or Blackman-Harris for lower sidelobes) before each FFT frame; use a one-pole leaky integrator (lambda * old + (1-lambda) * new) for temporal smoothing; implement peak hold with a configurable gravity/decay rate; map display x-positions logarithmically from FFT bin index to pixel; use dBFS on the y-axis with minDecibels/maxDecibels; prefer perceptual color maps (viridis, inferno) over jet for spectrograms; use correlation-based triggering for stable oscilloscope display; and optionally apply A-weighting before display to match perceived loudness. GPL-licensed projects (JUCE, audioMotion-analyzer, Essentia, Signalizer, GLava, spectroterm, Chord-Detector-and-Chromagram) carry copyleft obligations that propagate to derivative works. Permissive projects (Butterchurn/MIT, wavesurfer.js/BSD-3, spectrum-analyzer Rust/MIT, audio-oscilloscope/MIT, cava/MIT, corrscope/BSD-2, Meyda/MIT, SpectroGhost/MIT, spectro/MIT, constant-q-cpp/BSD, p5.sound/LGPL-2.1, Tone.js/MIT) can be used in commercial and closed-source products.

**Techniques**
- FFT Spectrum Bars — Linear binning: Web Audio AnalyserNode produces fftSize/2 uniformly-spaced bins from 0 Hz to Nyquist (sampleRate/2). Each bin width = sampleRate/fftSize Hz. For a 48 kHz source and fftSize=8192, each bin is ~5.86 Hz. Mapping bin index i to display pixel x requires log mapping: x = width * log(freq_i / minFreq) / log(maxFreq / minFreq) where freq_i = i * sampleRate / fftSize. Multiple FFT bins that map to the same pixel column should be combined by taking max (peak) rather than mean to avoid visual attenuation.
- FFT Windowing — Apply a window function to each analysis frame before FFT to prevent spectral leakage. Hann window at 50% overlap is the classic choice balancing frequency resolution and sidelobe rejection (~31 dB). Blackman-Harris (4-term, ~92 dB sidelobe rejection) is preferred when high dynamic range visualization is needed. Web Audio AnalyserNode applies Blackman window internally. For JUCE/manual FFT, use dsp::WindowingFunction<float> with Hann or BlackmanHarris type before calling performFrequencyOnlyForwardTransform().
- Temporal Smoothing — One-pole leaky integrator: smoothed[i] = lambda * smoothed[i] + (1-lambda) * raw[i]. Lambda near 1.0 = slow/smooth; near 0 = fast/jittery. Web Audio AnalyserNode exposes this as smoothingTimeConstant (0–1, default 0.8). For JUCE/manual, compute lambda from a time constant tau (ms): lambda = exp(-hopSize / (sampleRate * tau / 1000)). Separate attack and release lambdas (fast attack lambda_a ~0.1, slow release lambda_r ~0.9) produce more visually pleasing bars that snap up quickly and decay gradually.
- Peak Hold and Gravity Decay — Track a peak[] array alongside the smoothed spectrum. Each frame: if smoothed[i] > peak[i], immediately set peak[i] = smoothed[i] (zero attack). Otherwise, apply gravity: peak[i] -= decay_rate * delta_time (linear gravity in dB space) or peak[i] *= decay_factor per frame (exponential). Configurable HOLD_MS parameter holds the peak at maximum for a fixed duration before gravity begins. Typical values: hold = 500–2000 ms, decay = 6–12 dB/s.
- Log Frequency Axis and Octave Bands — Human hearing spans ~10 octaves (20 Hz to 20 kHz). Log frequency axis allocates equal screen width per octave, matching pitch perception. For octave-band analyzers: band k spans [f0 * 2^(k/N), f0 * 2^((k+1)/N)] for N bands per octave. Energy in each band = max or RMS of all FFT bins falling within the band. Common options: 1/1, 1/3, 1/6, 1/12, 1/24 octave bands. audioMotion-analyzer modes 1–8 implement 1/24 through 1/1 octave bands.
- Amplitude Scale (dB / dBFS) — Display magnitude in dBFS: dBFS = 20 * log10(|X[k]| / maxAmplitude). Web Audio getFloatFrequencyData() returns values in dBFS directly (typically -Inf to 0 dBFS). getByteFrequencyData() returns 0–255 scaled between minDecibels (default -100 dBFS) and maxDecibels (default 0 dBFS). Formula: dBFS = minDecibels + (byteValue/255) * (maxDecibels - minDecibels). Bar height on screen: barHeight = canvasHeight * (dBFS - minDecibels) / (maxDecibels - minDecibels).
- A-Weighting — Frequency-dependent perceptual correction. Formula (IEC 61672-1): W_A(f) = (12194^2 * f^4) / ((f^2 + 20.6^2)(f^2 + 12194^2) * sqrt((f^2 + 107.7^2)(f^2 + 737.9^2))). In dB: W_dB = 20*log10(W_A(f)) + 2.0. Apply by adding W_dB[k] to each FFT bin's dB value before display. The +2.0 normalization makes 0 dB at 1 kHz. audioMotion-analyzer exposes this as the 'A' weighting filter option. Alternatively implement as a biquad IIR filter in the time domain using bilinear transform of the analog prototype.
- Oscilloscope / Waveform — Read getByteTimeDomainData() or getFloatTimeDomainData() from AnalyserNode. Values are normalized amplitude -1.0 to 1.0 (float) or 0–255 (byte, offset binary: 128 = zero crossing). Draw as a connected polyline across canvas width. For stable display, implement zero-crossing trigger: scan the buffer for the first upward zero crossing (sample[i-1] < 128 && sample[i] >= 128 for byte data), offset the buffer start to that index. For polyphonic audio, use correlation-based triggering (corrscope algorithm): cross-correlate the current frame with the previous frame and find the lag that maximizes correlation.
- Lissajous / Vectorscope — Extract separate left and right channel time-domain buffers. Plot each sample pair (L[i], R[i]) as a point at canvas coordinates (x = L[i]*scale + cx, y = R[i]*scale + cy). Persistence is achieved by not clearing the canvas each frame (or clearing with a low-opacity black rectangle: fillRect with rgba(0,0,0,0.1)). Mono signal produces a diagonal line at 45 degrees. Wide stereo produces a cloud. Out-of-phase mono produces a horizontal line. For vectorscope mode (mid-side), plot M = (L+R)/sqrt(2) on the Y axis and S = (L-R)/sqrt(2) on the X axis.
- Spectrogram / Waterfall — Two standard canvas techniques: (1) Pixel shift: each frame, call ctx.drawImage(canvas, -1, 0) to shift content left by 1 pixel, then draw current FFT column on the rightmost pixel column. (2) Circular buffer texture (WebGL): write each FFT frame to a row of a floating-point texture using texSubImage2D(); scroll using mod() in the fragment shader to avoid CPU copies. Color mapping: convert dB magnitude to 0–1 range, look up in a 1D gradient texture (viridis, inferno preferred over jet for perceptual uniformity). For frequency axis, apply Mel scale mapping: mel = 2595 * log10(1 + f/700), or Bark: bark = 13*arctan(0.76*f/1000) + 3.5*arctan((f/7500)^2).
- Radial / Polar Spectrum — Take FFT magnitude array and map to polar coordinates: angle = 2*pi * i / numBins (or log-mapped angle for log frequency), radius = innerRadius + magnitude * scale. Draw lines from (cx + innerRadius*cos(angle), cy + innerRadius*sin(angle)) to (cx + outerRadius*cos(angle), cy + outerRadius*sin(angle)). For bar-style radial, compute bar width as the angular difference between adjacent bins. Mirror the spectrum by drawing it on both halves of the circle for aesthetic symmetry. audioMotion-analyzer implements this as radial: true mode.
- Constant-Q Transform (CQT) — Unlike FFT which has uniform Hz-per-bin resolution, CQT uses geometrically spaced bins: f_k = f_min * 2^(k/B) where B is bins per octave. Each bin has a different analysis window length: longer windows for low notes (better frequency resolution), shorter for high notes (better time resolution). Mirrors cochlear processing. The cannam/constant-q-cpp library implements this in C++ (BSD/MIT). librosa.cqt() in Python. foobar2000 foo_vis_spectrum_analyzer implements CQT, FFT, and SWIFT (Sliding Windowed Infinite Fourier Transform) in a single plugin.
- Chromagram — Fold CQT output into 12 pitch classes (C through B) by summing energy across octaves for each semitone. Result is a 12-element vector updated each frame. Display as 12 colored columns or a circular 'chroma wheel' with 30-degree segments. In Python: librosa.feature.chroma_cqt(y=audio, sr=sr) returns shape (12, T). In JavaScript: Meyda extracts 'chroma' feature. Key detection: correlate 12-bin chroma against Krumhansl-Schmuckler major/minor profiles for 24 candidate keys and take argmax.
- SWIFT Transform — Sliding Windowed Infinite Fourier Transform, also called the recursive DFT or Goertzel filter bank. Computes the DFT at specific frequencies using a recursive IIR approach, updating the complex output sample-by-sample with O(N) per frame rather than O(N log N). Particularly efficient for sparse frequency sets (e.g., musical note frequencies). Used in foobar2000 foo_vis_spectrum_analyzer for 'analog-style' spectrum display. Each Goertzel filter has a transfer function H(z) = 1/(1 - 2cos(2pi*f/fs)*z^-1 + z^-2).
- Overlap-Add / Hop Size — For smooth animation, FFT frames should overlap. At 44.1 kHz with fftSize=4096 (92 ms), 50% overlap (hop=2048) gives ~21 new frames/second; 75% overlap (hop=1024) gives ~43 frames/second. The Constant-Overlap-Add (COLA) constraint (satisfied by Hann at 50% or 75%) ensures all samples are weighted equally. Web Audio AnalyserNode internally uses 50% overlap. For manual STFT, store the last fftSize samples in a ring buffer, advance by hopSize each call.
- Perceptual Colormaps for Spectrograms — Prefer perceptually uniform colormaps: viridis (blue-green-yellow, monotonically increasing luminance), inferno (black-purple-orange-white), magma, plasma. Avoid jet (red-green-blue), which creates false contours and is not perceptually uniform. In Lab color space, interpolate between colormap control points for smooth gradients. The spectro project (MIT) uses quadratic Lab-space interpolation between 5 control points.

**Recommendations**
- Always apply a window function before FFT. Use Hann at 50% overlap as the safe default (satisfies COLA, ~31 dB sidelobe rejection). Upgrade to Blackman-Harris when you need >60 dB dynamic range (e.g., showing quiet room noise alongside loud peaks).
- Use log frequency axis for all spectrum bar displays. Map bin index to pixel x using: x = width * log(binFreq / minFreq) / log(maxFreq / minFreq). This allocates roughly equal visual space to each octave, matching musical and perceptual expectations.
- Use dBFS on the y axis, not linear amplitude. Typical range -90 to 0 dBFS for music. Set AnalyserNode.minDecibels = -90 and maxDecibels = 0. For professional metering, use -70 to +3 dBFS range.
- Implement separate attack and release smoothing rather than a single smoothingTimeConstant. Fast attack (lambda ~0.1, ~5 ms) makes bars respond immediately to transients; slow release (lambda ~0.9, ~200 ms) gives graceful decay. This matches professional hardware VU/PPM meter behavior.
- Use peak hold with gravity for spectrum bars: track a separate peak[] array, zero-attack (immediately adopt new max), hold for 500–2000 ms, then apply linear dB/s gravity. Values around 6 dB/s gravity and 1000 ms hold are common starting points.
- For oscilloscope/waveform, always implement a trigger. Zero-crossing trigger (find first upward zero-crossing in buffer) is adequate for single-tone signals. For complex polyphonic audio, use cross-correlation triggering (corrscope approach): maximize sum(current_frame[i] * prev_frame[i+lag]) over a small lag range.
- For spectrogram/waterfall, prefer WebGL texture approach over canvas pixel shift for performance. Write FFT data to a floating-point texture row using texSubImage2D(), scroll with mod() in fragment shader. Canvas pixel shift (getImageData/putImageData or drawImage offset) is acceptable for small canvas widths (<1000 px) but becomes CPU-bound at high resolutions.
- For chromagram visualization, compute CQT rather than STFT-based chroma. CQT chroma is more robust across octaves because the analysis window scales with frequency. Use librosa.feature.chroma_cqt() in Python or the cannam/constant-q-cpp Vamp plugin for offline analysis; use Meyda 'chroma' feature in the browser.
- Use perceptually uniform colormaps (viridis, inferno, magma) for spectrograms. Never use jet/rainbow colormaps in serious analysis tools — they introduce false contours and are inaccessible to colorblind users.
- For frequency scaling choices: log scale is best for general music visualization (matches pitch perception); Mel scale (m = 2595 * log10(1 + f/700)) is optimal for speech and vocal analysis; Bark scale (13*arctan(0.76f/1000) + 3.5*arctan((f/7500)^2)) reflects critical-band spacing; linear scale is only appropriate for scientific/engineering contexts where absolute Hz values matter.
- Avoid audioMotion-analyzer and JUCE in commercial closed-source products — both are AGPL-3.0. Use wavesurfer.js (BSD-3), Butterchurn (MIT), Meyda (MIT), cava (MIT), Tone.js (MIT), or corrscope (BSD-2) instead.
- Avoid Essentia ML models in any product (CC-BY-NC-ND 4.0 prohibits commercial use). The Essentia C++ core library also requires a commercial license from MTG/UPF for closed-source use.

**Pitfalls**
- GPL trap — audioMotion-analyzer is AGPL-3.0. Network-served web apps using it must release their source code. Confirmed from GitHub README. Do not embed in proprietary SaaS without a commercial license negotiation.
- GPL trap — JUCE is AGPL-3.0 for open-source use; a paid commercial license is required for closed-source plugins or apps. The dsp::FFT and WindowingFunction classes are inside JUCE, so they inherit this license.
- GPL trap — Essentia is AGPL-3.0 for non-commercial use; commercial use requires a license from MTG/UPF (email mtg-info@upf.edu). The pre-trained ML models bundled with Essentia are CC-BY-NC-ND 4.0, which prohibits commercial use outright.
- GPL trap — GLava (jarcode-foss/glava) is GPL-3.0 because it incorporates cava (MIT) and GLFFT (MIT) but re-licenses them under GPL. Cannot be linked into proprietary software.
- GPL trap — spectroterm (sparklost/spectroterm) is GPL-3.0.
- GPL trap — adamstark/Chord-Detector-and-Chromagram is GPL-3.0. Cannot be used in proprietary products.
- GPL trap — p5.js-sound (processing/p5.js-sound) is LGPL-2.1. LGPL permits dynamic linking without copyleft, but static linking or modification triggers copyleft. Verify linking approach before use in commercial products.
- Linear FFT bins at high frequencies — The Web Audio AnalyserNode produces linearly spaced bins. At 44.1 kHz with fftSize=2048, each bin is ~21.5 Hz wide. This means the top octave (11–22 kHz) is represented by ~512 bins while the bottom octave (20–40 Hz) gets fewer than 2 bins. Always remap to log frequency before display.
- smoothingTimeConstant applies only to getByteFrequencyData/getFloatFrequencyData — it does NOT affect getByteTimeDomainData or getFloatTimeDomainData (waveform). Apply separate temporal smoothing to waveform data if needed.
- Jet colormap in spectrograms creates false perceptual contours and fails for colorblind users (red-green colorblindness). Always prefer viridis, inferno, or magma.
- No window function = spectral leakage. Without windowing, a 1 kHz sine at -6 dBFS can contaminate adjacent bins with energy at -20 dBFS or higher, making the spectrum look smeared and noisy.
- Chromagram computed from STFT (chroma_stft) rather than CQT will alias across octave boundaries at high frequencies because STFT bin spacing is uniform in Hz, not in semitones. Prefer chroma_cqt or CQT-based chromagram for music analysis.
- FFT size too small gives poor low-frequency resolution. At 44.1 kHz, fftSize=512 gives 86 Hz/bin — C2 (65 Hz) and D2 (73 Hz) fall in the same bin and cannot be distinguished. Use fftSize >= 4096 for sub-100 Hz resolution.
- FFT size too large introduces temporal smearing. fftSize=32768 at 44.1 kHz is a ~742 ms window — fast transients (kick drum attack) appear smeared across time in spectrogram displays. Balance frequency resolution vs. time resolution for the target content.
- Peak hold without clamping to current bar level — if peak[] is initialized to a high value and no audio plays, the peaks remain frozen at maximum until actual audio exceeds them. Initialize peak[] to -Infinity at startup.
- CC-BY-NC-ND trap — Essentia's pre-trained ML models (for key, chord, mood, etc.) are licensed CC-BY-NC-ND 4.0. The NC (NonCommercial) clause prohibits any commercial use. The ND (NoDerivatives) clause prohibits fine-tuning or modification. Do not use these models in commercial products.

## 6. Web Creative-Coding / Rendering Frameworks for a Modular Visualizer Engine

Evaluated seven web rendering technologies as of June 2026 for use as the core engine of a modular visualizer supporting 2D, 3D, and raw-shader plugin modes at 60 fps. Three.js r184+ with WebGPURenderer and TSL is the clear primary engine choice: it is the only option that natively unifies 2D+3D+shader authoring, auto-compiles shaders to both GLSL and WGSL, ships a node-based post-processing pipeline (RenderPipeline) designed for modular pass composition, carries a zero-dependency MIT license, has 112K GitHub stars and 2.7M weekly npm downloads, and has full production-grade WebGPU support since r171 (September 2025) with automatic WebGL2 fallback. **Techniques**
- Three.js TSL (Three.js Shading Language): node-based JS shader graphs that compile to GLSL or WGSL transparently. Plugin authors write one shader in TSL; it runs on WebGL2 and WebGPU without modification. Introduced in r166, production-hardened in r184.
- Three.js RenderPipeline (WebGPU path): composable graph of render nodes replacing the linear EffectComposer chain. Each plugin contributes a node (SceneRenderNode, BloomNode, custom ComputeNode). Built for WebGPURenderer with automatic WebGL2 fallback.
- Three.js EffectComposer + pmndrs/postprocessing (WebGL2 path): mature pass-based post-processing compatible with Three.js 0.157-0.176. Multiple effects in one EffectPass merge into a single shader draw. Use as the fallback pipeline when WebGPU is unavailable.
- Shared WebGL context pattern (Three.js + PixiJS): Three.js creates the WebGLRenderingContext; PixiJS receives it via context option. Rendering sequence: three.render() -> three.resetState() -> pixi.resetState() -> pixi.render() -> requestAnimationFrame. Enable stencil:true in Three.js for PixiJS masks. Keeps a single GPU context for both 2D and 3D layers.
- WebGPU compute shaders for non-rendering workloads: particle simulation, audio FFT, physics, ML inference. Compute pipelines run independently of the render pipeline and read results back to JS or feed directly into vertex/fragment shaders via storage buffers. Available in Three.js via TSL compute nodes or raw GPUDevice.createComputePipeline.
- OGL as an ultra-lean shader plugin host: OGL's Renderer + Program + Geometry pattern wraps a WebGL2 context in ~8 KB. Plugin modes that only need fullscreen fragment shaders (audio reactive visuals, image effects) can instantiate OGL without pulling the full Three.js runtime, then composite the result as a texture into the main Three.js scene.
- regl for data-visualization plugin modes: functional command pattern (const drawPoints = regl({ frag, vert, attributes, count }); drawPoints()) makes data-driven GPU rendering straightforward. Best for 2D scientific visualizations (scatter plots, heatmaps, graph layouts) as a plugin type distinct from the 3D scene graph.
- Feature detection + graceful degradation: WebGPU.isAvailable() from 'three/addons/capabilities/WebGPU.js' gates WebGPURenderer instantiation; fall back to WebGLRenderer. At plugin level, navigator.gpu !== undefined guards WebGPU compute paths. Always ship a WebGL2 fallback for the ~18-29% of users without WebGPU.
- Tree-shaking discipline for bundle control: import only used Three.js classes (never import * from 'three'). Use dynamic import() to load plugin mode bundles on demand. OGL's ES module design enables per-class tree-shaking down to single-digit KB for minimal modes.
- p5.js isolation pattern for education/demo modes: load p5.js in a sandboxed iframe or Web Worker context. Never co-render p5 on the same canvas as the primary Three.js scene. This avoids p5's WebGL state interference and keeps its ~800 KB bundle out of the main chunk.

**Recommendations**
- PRIMARY ENGINE: Three.js (MIT, r184+). Use WebGPURenderer with automatic WebGL2 fallback as the single rendering context owner. Write all internal shaders in TSL for forward compatibility. Build the plugin API around TSL nodes and the RenderPipeline graph so plugin authors register composable passes, not raw draw calls.
- 2D OVERLAY LAYER: PixiJS v8 (MIT). Use the shared-context pattern to composite 2D UI, sprite sheets, and particle HUD elements over the Three.js 3D scene. Reserve PixiJS for strictly 2D content; do not use it for 3D geometry.
- ULTRA-LEAN SHADER PLUGIN TYPE: OGL (Unlicense). Define a plugin interface that accepts an OGL Renderer + Program bundle. These plugins produce a fullscreen texture composited into the Three.js RenderPipeline as a TextureNode. Total plugin overhead: <25 KB before tree-shaking.
- DATA-VISUALIZATION PLUGIN TYPE: regl (MIT). For scientific/data plugin modes (scatter, graph, heatmap), regl's functional draw command pattern is more ergonomic than Three.js geometry management. Plugins declare a regl context sharing the existing WebGL context via regl({ gl: threeRenderer.getContext() }).
- EDUCATION/BEGINNER PLUGIN MODE: p5.js (LGPL-2.1), isolated in a sandboxed iframe. Expose a message-bus API from the main engine. Do not co-render. Flag LGPL-2.1 to legal if the host app is distributed as a binary (e.g., Electron): LGPL requires dynamic linking or source disclosure of modifications to p5 itself; the user sketch code is unaffected.
- SYNTHESIZER UX REFERENCE ONLY: Study Hydra's feedback-framebuffer metaphor and o0..o3 output-routing UX. Implement an equivalent natively using Three.js WebGLRenderTarget ping-pong buffers or WebGPU storage textures. DO NOT import hydra-synth as a dependency: AGPL-3.0 copyleft would require releasing the entire visualizer codebase under AGPL-3.0.
- RAW WebGPU ESCAPE HATCH: Provide a 'native compute' plugin type that receives a GPUDevice handle. Plugin authors write WGSL compute shaders directly. Gate this type with navigator.gpu feature detection. This enables ML inference, physics simulation, and audio FFT plugins with maximum throughput and zero framework overhead.
- BUNDLE ARCHITECTURE: Lazy-load plugin bundles with dynamic import(). Core engine (Three.js WebGPURenderer + minimal scene graph): aim for <200 KB gzip on first load. Each plugin type loads its own secondary library (OGL, regl, PixiJS) only when that plugin mode is activated. This avoids shipping 168 KB (Three.js) + 71 MB (PixiJS full npm directory) on initial load — use the tree-shaken import paths.
- WEBGPU FALLBACK STRATEGY: Ship all TSL shaders with the Three.js default; Three.js compiles to WGSL on WebGPU and GLSL on WebGL2 automatically since r171. For raw compute plugins, provide a CPU/WASM fallback path. Target Chrome 113+, Edge 113+, Firefox 147+, Safari 26+ for WebGPU; treat everything else as WebGL2.

**Pitfalls**
- AGPL LICENSE TRAP — Hydra / hydra-synth: Both the main hydra repo and the hydra-synth npm package are AGPL-3.0. Importing hydra-synth as a JS module in a web app served over a network triggers the AGPL network-use clause, requiring the entire app source to be published under AGPL-3.0. This is a hard block for any closed-source or commercially licensed product. Verified from https://github.com/hydra-synth/hydra-synth (AGPL-3.0 license badge). Do not embed; re-implement the UX concepts instead.
- LGPL-2.1 for p5.js — not GPL but not fully permissive: p5.js uses LGPL-2.1, not MIT. For web use (linking as a JS module), the LGPL's dynamic-linking exception applies and the user's sketch code stays proprietary. However, in Electron/Node.js binary distribution contexts, LGPL may require shipping p5.js as a separately replaceable dynamic module. Modifications to p5.js itself must be released under LGPL-2.1. Verify with legal counsel for binary/desktop distribution.
- p5.js WebGPU renderer is NOT a performance option: The p5.strands WebGPU renderer (introduced in p5.js 2.1-2.2) explicitly does not aim to be a high-performance renderer (confirmed in GitHub issue #8471). p5.js 2.0 beta also regressed WebGL mode from 60fps to ~20fps in tested scenarios. Do not use p5.js in any performance-critical rendering path.
- regl has no native WebGL2 or WebGPU support: GitHub issues #378 (2016) and #561 (2020) requesting WebGL2 support remain open and unimplemented. Community workarounds exist (regl-webgl2-compat) but are not official. regl cannot use WebGL2 instanced arrays, transform feedback, MRT, or SSBO extensions natively. Do not use regl for effects requiring WebGL2-only features.
- OGL has no WebGPU support and low maintenance velocity: OGL v1.0.11 was last published approximately 1 year ago. The companion oframe/ogpu repo has only 1 commit and 31 stars — it is a proof-of-concept, not a production library. OGL is WebGL-only. Do not plan a WebGPU upgrade path through OGL.
- Three.js EffectComposer is WebGL-only and does NOT work with WebGPURenderer: The legacy EffectComposer targets WebGLRenderer. For WebGPU rendering paths, use the new RenderPipeline / node-based post-processing system. Mixing EffectComposer passes with WebGPURenderer will silently fall back or error. pmndrs/postprocessing also lists compatibility only up to Three.js 0.176.
- PixiJS WebGPU renderer is not recommended for production mobile as of v8.19.0: PixiJS documentation explicitly states WebGPU on mobile has inconsistencies and recommends WebGL2 renderer for production mobile targets. The transient flag optimization in v8.19.0 helps mobile MSAA but does not resolve all driver inconsistencies.
- Shared WebGL context between Three.js and PixiJS requires strict state reset discipline: Both renderers mutate shared WebGL state (VAO bindings, framebuffer bindings, program state). Missing a resetState() call on either side causes rendering artifacts. Enable stencil:true in the Three.js renderer constructor for PixiJS masks to work. Resources (textures, buffers) cannot be shared between the two — PixiJS textures are not directly usable as Three.js textures.
- WebGPU ~82% global desktop, ~71% mobile: as of caniuse June 2026, Firefox desktop shows WebGPU as disabled by default in stable (flag-only), and Firefox for Android, UC Browser, and standard Android browser have no support. Linux WebGPU support is driver-dependent and inconsistent. Always ship a WebGL2 fallback; do not assume WebGPU universality even in 2026.
- Three.js TSL syntax is verbose: TSL requires method-chaining (.add(), .mul(), .mix()) instead of infix operators. This trades ergonomics for cross-platform compilation and IDE type safety. Plugin authors used to raw GLSL will need onboarding. Raw GLSL strings via RawShaderMaterial or onBeforeCompile remain available as an escape hatch for WebGL-only plugins, but these will not run under WebGPURenderer without porting to TSL or WGSL.

## 7. Audio Feature Extraction for Music Visualizers (Browser Realtime + Offline Python)

Two-path architecture is standard: a browser realtime path using Web Audio AnalyserNode + Meyda.js for per-frame features, and an offline Python path (librosa primary, essentia/madmom for high-quality beat/key) that pre-analyses the full track and exports a timestamped JSON sidecar. The browser loads the sidecar at playback time for features that are too expensive or inaccurate to compute in realtime (global BPM, musical key, downbeats). Meyda (MIT) is the only permissively-licensed JS library covering the full standard MIR feature set; Essentia.js (AGPL-3.0) covers more features but is a license trap for commercial products. aubiojs (MIT wrapper, GPL core via emscripten) and aubio Python (GPL-3.0) require careful GPL compliance review. madmom pre-trained models carry CC-BY-NC-SA 4.0, blocking all commercial use. beat_this (MIT, ISMIR 2024) is the modern replacement for madmom beat tracking with no license restrictions.

**Techniques**

- Spectral Rolloff: Meyda 'spectralRolloff' (frequency below which 99% of energy falls). Offline: librosa.feature.spectral_rolloff(roll_percent=0.85).
- Onset Detection: No native browser API. Meyda 'spectralFlux' + manual peak-picking provides basic onset detection. aubiojs provides proper onset detection with multiple algorithms in browser. Offline: librosa.onset.onset_detect() (multiple methods: energy, spectral flux, superflux), aubio OnsetDetection, essentia OnsetDetection algorithm.
- Beat/Tempo (BPM) Tracking: No built-in method in AnalyserNode or Meyda. aubiojs Tempo class provides BPM + beat tick in browser. Essentia.js EssentiaExtractor provides BPM and beat positions (AGPL). Offline: librosa.beat.beat_track() (returns BPM + beat frame positions, converts to seconds via librosa.frames_to_time), beat_this (MIT, SOTA deep-learning beats+downbeats), essentia RhythmExtractor2013 (AGPL), madmom DBNBeatTracker (CC-BY-NC-SA).
- Key/Chroma: Meyda 'chroma' (12-element chroma vector per frame, real-time). Essentia.js HPCP extractor (AGPL). No realtime key classifier in MIT-licensed JS. Offline: librosa.feature.chroma_cqt()/chroma_stft() for chroma; key detection via Krumhansl-Schmuckler profile matching over aggregated chroma (not a single librosa function, requires manual implementation or essentia Key algorithm). librosa.feature.tonnetz() for tonal centroid. essentia Key algorithm + HPCP (AGPL). madmom key classification (CC-BY-NC-SA).
- MFCC: Meyda 'mfcc' (13 coefficients, Mel filterbank, real-time per frame). Essentia.js MFCC (AGPL). Offline: librosa.feature.mfcc(y, sr, n_mfcc=13) returns shape (n_mfcc, T). aubio MFCC (GPL). essentia MFCC algorithm (AGPL).
- Offline Pre-Analysis Pipeline Pattern: Run librosa + beat_this offline on the full audio file; compute per-frame features (chroma, MFCC, RMS, spectral centroid) at a reduced hop_length; detect onsets and beats; run Krumhansl-Schmuckler key detection on aggregated chroma; serialize all results with timestamps (librosa.frames_to_time) to a JSON sidecar file. At playback in browser, load the JSON, index by audio.currentTime, drive visualizer state from pre-computed data instead of realtime extraction.

**Recommendations**
- Minimal robust JS feature set for a visualizer: (1) AnalyserNode for raw FFT spectrum visualization (no extra dependency), (2) Meyda.js for: rms, energy, spectralCentroid, spectralFlux, spectralRolloff, chroma, mfcc — all MIT and realtime-capable, (3) aubiojs for beat tick and BPM if realtime BPM display is required (GPL compliance check needed). This covers 90% of visualizer needs with two dependencies.
- Do NOT use Essentia.js for commercial visualizers without purchasing a commercial license from MTG/UPF. The AGPL-3.0 requires open-sourcing any web app that uses it or obtaining a paid license.
- For offline pre-analysis pipeline (recommended for any feature requiring global track context): use librosa (ISC) as the primary library — covers FFT, RMS, spectral centroid/rolloff/flatness/bandwidth, onset detection, beat tracking, chroma, MFCC, tempo. Add beat_this (MIT) for higher-accuracy beat and downbeat positions. Avoid madmom in commercial projects (CC-BY-NC-SA on models). Avoid essentia Python unless AGPL compliance is acceptable.
- Pre-compute global BPM, musical key, beat positions, and downbeat positions offline. These cannot be computed accurately in realtime from a short analysis window. Serve them as a JSON sidecar. At playback, sync visualizer state transitions (key changes, measure boundaries) to audio.currentTime against the sidecar's timestamp array.
- MFCC in a visualizer context: 13 coefficients per frame is sufficient for timbre coloring or genre-derived palette mapping. In realtime, Meyda mfcc is accurate enough. Offline, librosa.feature.mfcc() with delta() coefficients gives richer timbre trajectories for animations.
- Spectral flux from Meyda ('spectralFlux') is the simplest onset strength proxy for driving real-time reactive animations (particle bursts, beat flash). Use a rolling buffer and peak threshold rather than raw flux values.
- Tone.js Analyser/FFT is appropriate only if you are already using Tone.js for audio scheduling/synthesis. It wraps AnalyserNode with no added feature extraction. Do not add Tone.js solely for analysis — use Meyda instead.
- For chroma-based key display in realtime: accumulate Meyda 'chroma' vectors over a 2–4 second sliding window, normalize, and run a simple Krumhansl-Schmuckler dot-product against 24 key profiles. This is implementable in ~50 lines of JS without any additional library and gives reasonable key tracking. For global key, always use the offline pre-computed result.

**Pitfalls**
- Essentia.js AGPL-3.0 trap: Any web application that includes Essentia.js WASM bundle and is served over a network is subject to AGPL network-service copyleft. You must either open-source your entire application or obtain a commercial license from MTG/UPF. This is commonly overlooked because the npm package installs without warning.
- aubio GPL-3.0 trap: aubio (Python and C library) is GPL-3.0. aubiojs wraps it via emscripten — the resulting WASM blob distributes GPL-3.0 code. Any application bundle containing aubiojs must comply with GPL-3.0 (provide source, allow modification and redistribution). The aubiojs wrapper's own MIT license does not override the GPL of the compiled aubio code it contains.
- madmom CC-BY-NC-SA 4.0 on pre-trained models: The model weights (which are required for the best beat/downbeat trackers like DBNBeatTracker) are CC-BY-NC-SA 4.0. Non-commercial use only. ShareAlike means any derivative work must use the same license. Do not use madmom models in commercial products.
- AnalyserNode only provides smoothed log-magnitude (dB) FFT data, not raw complex FFT. Phase information is lost. There is no access to spectral centroid, flux, chroma, MFCC, or any higher-level feature natively. All derived features require additional library code (e.g. Meyda).
- Meyda v5.6.3 has no beat tracking or global tempo estimation. Spectral flux per frame can drive reactive animation but cannot determine BPM without a separate beat tracking algorithm (autocorrelation, comb filter, or neural network).
- Essentia.js npm package (v0.1.3) is 4 years stale on npm. The GitHub master branch may have newer builds but they are not published to npm. Production use requires building from source or using unpublished artifacts.
- aubiojs (v0.2.1, Nov 2022) has limited maintenance (10 open issues, 0 PRs merged recently). The API surface covers only tempo and pitch — onset detection from aubio C library is not exposed in the JS wrapper.
- librosa does not include a built-in key detection function. Chroma extraction (chroma_cqt, chroma_stft) is provided, but mapping chroma to a musical key requires manual Krumhansl-Schmuckler profile matching or use of essentia Key algorithm (AGPL).
- madmom requires Python 3 and specific NumPy/SciPy versions. It is not actively maintained as of 2024 (last release v0.16.1 in 2018). Use beat_this (MIT, 2024) instead for beat and downbeat tracking.
- Realtime key detection in the browser from a short analysis window is inherently unreliable. Key identification requires seeing enough harmonic content across the full song. Always prefer offline pre-computed global key and display it as a static label, not a per-frame realtime update.
- Tone.js version confusion: npm stable is 14.7.39; there is a 15.x series (15.1.22 documented in tonejs.github.io) that may not be the default npm install. Pin versions explicitly.

## 8. Music Visualizer Distribution & Video Export Pipeline

A music channel visualizer has three viable distribution paths: (1) instant zero-install static web app on Netlify/Vercel/GitHub Pages; (2) desktop-wrapped native app via Tauri (preferred) or Electron; (3) offline/headless server-side render to a high-quality MP4 or ProRes MOV for YouTube. The gold-standard offline export pipeline is: drive a headless Chromium instance with HeadlessExperimental.beginFrame to pull deterministic frames at exact frame timestamps, pipe raw PNG frames + a separately rendered WAV (via OfflineAudioContext) into native ffmpeg (not ffmpeg.wasm) to produce ProRes 422 HQ or H.264 MP4. This guarantees perfect A/V sync because the browser clock is fully virtualized—time never advances until you request the next frame—and ffmpeg's mux step aligns the audio stream by sample, not by wall clock. WebCodecs VideoEncoder is the modern in-browser alternative for smaller files but requires a separate muxer (mp4box.js) and cannot yet produce ProRes.

**Techniques**
- Static web app deployment (Vite build -> Netlify/Vercel/GitHub Pages): zero install, shareable URL, works in any modern browser. Build with 'vite build', set base path for subfolder deploys. Ship as a single-page app with all assets hashed and CDN-cached.
- CCapture.js fixed-framerate capture: patches requestAnimationFrame to call your render function once per logical frame regardless of wall time. Outputs PNG/JPEG sequences or WebM. Audio NOT included — must mux externally. Suitable only for visual-only frame export within the browser.
- WebCodecs VideoEncoder in-browser pipeline: encode canvas frames via VideoEncoder (H.264/VP9/AV1) + mux with mp4box.js. Run in a Web Worker with OffscreenCanvas to avoid blocking the UI. AudioEncoder encodes PCM from OfflineAudioContext. This produces a full MP4 client-side with no WASM overhead. ProRes not supported.
- ffmpeg.wasm in-browser muxing: loads ffmpeg as WebAssembly, writes PNG frames into an in-memory filesystem, calls ffmpeg to produce MP4. 25-128x slower than native; no GPU acceleration; no ProRes; 2 GB browser memory cap. Use only for short clips or previews.
- HeadlessExperimental.beginFrame deterministic rendering (WebVideoCreator / headless-screen-recorder): launches Chrome with --deterministic-mode --disable-gpu --run-all-compositor-stages-before-draw flags, virtualizes Date/performance.now()/setTimeout/rAF, calls beginFrame once per logical frame to get a PNG screenshot. Frame computation time is irrelevant — a 2-second compute frame still produces a clean 60fps output. Works on Linux and Windows; unreliable on macOS.
- Clock-faking / time virtualization (Replit approach): injects JS to replace all timing APIs (setTimeout, setInterval, requestAnimationFrame, Date, performance.now) with a controlled virtual clock. Advances clock by exactly 1000/fps ms per frame, fires all callbacks, captures screenshot. Enables rendering of arbitrary HTML5 pages (not just framework-specific ones) deterministically.
- Remotion React-component rendering: each frame is a function of a frame-index prop, making rendering embarrassingly parallel. Headless Chromium renders each frame; ffmpeg stitches the output. ProRes supported via proresProfile. Audio sync handled natively. License: free for individuals / small teams.
- OfflineAudioContext for deterministic audio export: constructs full Web Audio graph, calls startRendering() to get an AudioBuffer synchronously-resolved promise. Float32 PCM samples are written to a WAV blob. This WAV is the audio input to ffmpeg for the final mux, guaranteeing sample-accurate sync.
- Native ffmpeg PNG-sequence-to-ProRes mux: 'ffmpeg -f image2 -framerate 60 -i frame_%06d.png -i audio.wav -c:v prores_ks -profile:v 3 -c:a pcm_s16le output.mov'. Profile 3 = ProRes 422 HQ, profile 4 = ProRes 4444. Audio is sample-accurate because both inputs start at t=0 and ffmpeg timestamps based on sample count, not wall clock.
- Native ffmpeg PNG-sequence-to-H.264 mux for YouTube: 'ffmpeg -f image2 -framerate 60 -i frame_%06d.png -i audio.wav -c:v libx264 -crf 18 -pix_fmt yuv420p -movflags +faststart -c:a aac -b:a 320k output.mp4'. -movflags +faststart moves the moov atom to the front for streaming. YouTube recommends H.264 High Profile, yuv420p, 48kHz AAC.
- Tauri desktop wrapping: replace Electron in distribution builds. Use Tauri 2.x for cross-platform (Win/Mac/Linux/iOS/Android) < 10 MB installer. Expose a Rust sidecar to invoke native ffmpeg binary for export, avoiding WASM overhead. The web frontend stays as-is; no code changes needed.
- Electron desktop wrapping with in-app export: bundle a headless Chromium (already present) and use electron's BrowserWindow in offscreen rendering mode to capture frames via CDP. More complex than Tauri but enables fully in-app export without a separate server. MIT license.

**Recommendations**
- Distribution as static web app is the fastest path to audience: build with Vite, deploy to Netlify or Vercel (both free tiers). Users open a URL, no install. Enable SharedArrayBuffer headers (COOP/COEP) on the host if using ffmpeg.wasm multi-thread mode.
- For desktop distribution, use Tauri (MIT/Apache-2.0, dual-licensed) over Electron. Tauri produces installers under 10 MB, uses 30-50 MB RAM vs Electron's 150-300 MB, and the Rust permission bridge improves security. Tauri 2.x also targets iOS/Android from the same codebase.
- For the offline/headless MP4 export pipeline, use this concrete stack: (1) WebVideoCreator (Apache-2.0) or a custom Node.js script using Puppeteer + HeadlessExperimental.beginFrame to pull PNG frames deterministically; (2) OfflineAudioContext to render the full audio mix to WAV; (3) native ffmpeg (LGPL/GPL build) to mux frames + WAV into ProRes 422 HQ MOV or H.264 MP4. This is the only approach that guarantees perfect A/V sync regardless of visualizer complexity.
- For YouTube uploads, produce two outputs from the same PNG+WAV inputs: a ProRes 422 HQ MOV as the master archive, and an H.264 MP4 (CRF 18, yuv420p, AAC 320k, +faststart) for direct upload. YouTube re-encodes everything so H.264 at high bitrate is sufficient; ProRes upload is only marginally better and much larger.
- For in-browser export of short clips (under 3 minutes, 1080p), use the WebCodecs VideoEncoder + mp4box.js (BSD-3-Clause) pipeline in a Web Worker. This avoids ffmpeg.wasm's 25-128x speed penalty. Pair with OfflineAudioContext for audio and AudioEncoder for the audio track. This is the best browser-native option for 2025 given Safari 26.0+ now supports WebCodecs.
- Avoid CCapture.js for new projects: last updated 2016, no audio track support, WebM output has browser compatibility constraints (requires image/webp support). Use canvas-capture (MIT) as a drop-in modern wrapper if you need a quick prototype, but plan to replace it with WebCodecs for production.
- For the audio sync guarantee: always export audio via OfflineAudioContext.startRendering() to a WAV file first, completely separately from the video frame capture. Never attempt to sync via wall-clock timing between an AudioContext and a canvas animation loop — clock drift is unavoidable. The WAV and the PNG sequence share the same sample rate and frame count; ffmpeg aligns them by timestamps at mux time.
- On macOS, do NOT rely on HeadlessExperimental.beginFrame for deterministic rendering — Chrome's --deterministic-mode flags do not reliably work on macOS. For macOS headless export, use Remotion (which uses a different frame-index-prop mechanism) or run the Linux/Windows export path in a Docker container (e.g., node:20-bullseye with Chrome installed).
- Check Remotion's license before using it commercially: free for individuals and companies with up to 3 employees. Organizations above that threshold need a paid Company License (starts at $100/month for server rendering). For a solo music channel creator, Remotion is free.

**Pitfalls**
- GPL trap in ffmpeg libx264: native ffmpeg built with libx264 is GPL-licensed, which propagates to any binary that statically links it. For a commercial product, either distribute ffmpeg as a separate binary (dynamic linking), use a platform encoder (h264_nvenc on NVIDIA, h264_videotoolbox on macOS), or use the libx264-licensed LGPL build without GPL extras. The prores_ks encoder is LGPL and safe.
- Remotion is NOT MIT: it has a custom license requiring a paid Company License for for-profit organizations with more than 3 employees. The source is on GitHub but is not freely usable commercially at scale. Verify eligibility at remotion.pro/license before building a product on it.
- ffmpeg.wasm is 25-128x slower than native and cannot use GPU acceleration. A 60-second 1080p60 music video could take 30-90 minutes to render in-browser via WASM. Do not use ffmpeg.wasm for production YouTube exports; use it only for short previews or development.
- HeadlessExperimental.beginFrame is documented as experimental and platform-restricted: works on Linux and Windows headless Chrome only. macOS Chrome does not reliably honor deterministic-mode flags. CI/CD pipelines for export should run on Linux (Docker).
- WebCodecs ProRes is not available in browsers: the prores codec is a platform-native encoder (Apple only on macOS/iOS) and is not exposed via the WebCodecs API in any browser. ProRes can only be produced by native ffmpeg. Do not attempt to use VideoEncoder for ProRes.
- CCapture.js audio sync is not a solved problem: GitHub issue #65 (open) documents that audio is NOT recorded with video frames. The currentTime patch only affects playback position during capture; the audio track is not embedded. Any music visualizer relying on CCapture.js must mux audio separately in a post-processing step.
- canvas-capture's last npm publish was ~3 years ago and it depends on a fork of CCapture.js (npm-fix branch) due to import issues with the upstream package. This creates a fragile dependency chain for a production music visualizer.
- SharedArrayBuffer (required for ffmpeg.wasm multi-thread) needs COOP/COEP HTTP headers: Cross-Origin-Opener-Policy: same-origin and Cross-Origin-Embedder-Policy: require-corp. These headers break embedding in iframes and can interfere with third-party analytics/ad scripts. GitHub Pages does not set these headers by default — use Netlify or Vercel with custom headers config.
- mp4box.js (BSD-3-Clause) is not the same as the main GPAC project (LGPL). The npm package mp4box is BSD-3, safe to use. Do not confuse it with the broader GPAC desktop application which is LGPL and has different distribution obligations.
- Tauri cannot use HeadlessExperimental.beginFrame via its own WebView for export: the OS-native WebView (WKWebView on macOS, WebView2 on Windows) does not expose the Chrome DevTools Protocol's HeadlessExperimental domain. For in-app headless export in a Tauri app, bundle a separate Puppeteer/Node sidecar that launches a full Chromium instance, or invoke native ffmpeg directly with pre-captured frames.

## 9. open-source music visualizer landscape analysis

Survey of 12+ active open-source and commercial music/audio visualizer projects as of mid-2026. The field is dominated by two paradigms: (1) Milkdrop-preset-driven systems (projectM, butterchurn, MilkDrop3) that require users to write equations in a proprietary format, and (2) single-purpose video-export tools (Wav2Bar, Astrofox) aimed at content creators. No currently-maintained open-source project combines a permissive license, a true component/scene-graph architecture that mirrors game/demo-scene idioms, a first-class contributor onboarding path, cross-platform real-time rendering, and a documented plugin API that a web or C++ developer can implement without learning a 20-year-old preset DSL. **Techniques**
- Milkdrop preset DSL (ns-eel2 expression evaluator + per-pixel/per-vertex equations): used by projectM, butterchurn, MilkDrop3, BeatDrop. Achieves infinite visual variety but requires learning a proprietary non-standard language.
- ISF (Interactive Shader Format): JSON-wrapped GLSL shaders with declared audio-reactive uniforms. Used by Synesthesia, modV. Portable across ISF-compatible hosts, can import from Shadertoy with adaptation.
- GLSL/HLSL fragment shaders as primary visual primitive: used by Synesthesia, MilkDrop3, Bonzomatic, Pulse. Direct GPU access, maximum performance, requires shader knowledge.
- FFT/spectrum analysis with beat detection: universal across all projects. FFTW (C/C++ projects), Web Audio API AnalyserNode (browser projects), Meyda (JS feature extraction). Beat detection quality varies widely.
- Audio stem separation for per-instrument reactivity: notable by near-total absence. Only discussed theoretically in HN comments about projectM. No visualizer ships this as a standard feature.
- Node/scene-graph visual composition: VSXu (archived) used this. modV approximates it. No active project has a first-class node editor for composing visual effects.
- WebGL2 / WebGPU rendering: butterchurn uses WebGL2. modV uses WebGL+Canvas. No project has shipped WebGPU-based visualizations at production quality yet.
- Native plugin ABI (.so/.dll): Pulse Visualizer has experimental plugin .so/.dll support. No other surveyed project has a documented, stable native plugin API.
- Video export pipeline: Wav2Bar, Astrofox, Wav2Bar-reborn via FFmpeg. All other projects are real-time display only.
- MIDI/OSC audio-reactive parameter control: Synesthesia (best implementation), modV (ISF uniforms), Plane9 (limited). Most open-source projects have no MIDI/OSC support.
- Layer/compositing model: Astrofox has a layer compositor. Synesthesia has scene stacking. Plane9 has scene combination. No open-source real-time visualizer has a documented compositor API.
- Reactive audio feature routing (map FFT band X to effect parameter Y): Synesthesia does this well via its audio-reactivity controls. modV approximates this. No open-source project has a clean declarative routing graph.

**Recommendations**
- Use MIT or Apache-2.0 license exclusively. The landscape is littered with GPL/LGPL/AGPL traps (projectM LGPL-2.1, Wav2Bar GPL-3.0, audioMotion-analyzer AGPL-3.0, Pulse GPL-3.0, VSXu GPL/LGPL archived) that block commercial embedding and derivative works. A permissive license is the single largest differentiator for adoption.
- Adopt ISF (Interactive Shader Format) as the primary effect primitive instead of Milkdrop DSL. ISF is GLSL-based, standard, supported by Synesthesia and modV, and allows import from the large Shadertoy ecosystem. This lowers the barrier to contributing new effects dramatically compared to learning ns-eel2.
- Build a first-class component/module graph as the core architecture, explicitly inspired by game engine entity-component systems. Each visual element (particle system, waveform, spectrum bars, shader quad, post-process pass) is an independently reloadable module with declared audio-input ports and render-output ports. This is what VSXu proved worked but abandoned, and what modV approximates without formalizing.
- Ship a stable documented plugin API from day 1, not as an afterthought. Pulse Visualizer marks its plugin system 'experimental'; projectM has no plugin API for new effect types; butterchurn has no extension points. A versioned plugin interface (similar to a DAW VST concept but for visual effects) is the contribution model that enables a community.
- Target WebGPU as the primary rendering backend for the web build and Vulkan/Metal via wgpu-rs or Dawn for the native build. All incumbent open-source projects are on WebGL2, OpenGL 3.x, or Direct3D 11. WebGPU enables compute shaders for audio analysis on GPU, modern bindless resources, and cross-platform native+web parity.
- Implement proper audio analysis beyond FFT: onset detection, RMS per band, spectral centroid, chroma features, and optionally real-time stem separation via a WASM-compiled model (e.g., Demucs lite). This addresses the gap explicitly identified in HN comments about projectM — that all current visualizers only see rhythm, not harmonic content or individual instruments.
- Provide a code-first contributor path alongside a GUI path. modV is GUI-first; butterchurn requires preset files; projectM requires C++. A new project should allow a contributor to write a TypeScript or Rust module implementing a 3-method interface (init, render, destroy) and see it appear in the compositor immediately, with no build step for the happy path.
- Build explicit game/demoscene idioms into the scene format: timeline-based automation curves (like a game engine animator), trigger events on beat, transition effects between scenes with configurable easing, and a scene scripting layer (Lua or WGSL compute). Bonzomatic proves demosceners want live-code flow; Synesthesia proves VJs want timeline+MIDI; no open-source project serves both.
- Cross-platform from the start: web (WASM+WebGPU), Windows, macOS, Linux. Plane9 is Windows-only. MilkDrop3 is Windows-only. projectM frontends are development previews. Synesthesia is Windows/Mac only. The only fully cross-platform open-source option is butterchurn (browser) or CAVA (terminal). A desktop+web project with a single codebase is unoccupied.
- Publish quantitative audio-reactivity quality benchmarks. No existing project measures or documents how accurately their beat detection tracks onset times, how low their audio-to-visual latency is, or how they perform on non-4/4 time signatures. Publishing this data would immediately differentiate a new project in search results and developer discussions.

**Pitfalls**
- GPL/LGPL license lock-in: choosing GPL-3.0 (Pulse, Wav2Bar original) or LGPL-2.1 (projectM) will immediately replicate the same adoption barriers those projects face. Any commercial game studio, streaming service, or content platform cannot embed GPL code.
- AGPL network copyleft trap: audioMotion-analyzer's AGPL-3.0 means any web app serving it over a network must open-source the entire server. This is why audioMotion-analyzer has 918 stars but minimal commercial adoption despite excellent technical quality.
- Milkdrop preset format lock-in: building compatibility with .milk presets seems attractive (access to 73k+ presets) but chains the architecture to ns-eel2 expression parsing and HLSL-to-GLSL transpilation that breaks 5-15% of presets. projectM has been fighting this for 20 years. Treat .milk import as an optional compatibility layer, not a core format.
- Electron performance ceiling: modV and Astrofox chose Electron. Electron's compositor adds latency, caps frame rates on some platforms, and creates a large binary size. Tauri (wav2bar-reborn) is better but Rust+WebView limits GPU API access. For a performance-critical visualizer, Electron/Tauri should not be the primary rendering path.
- Abandoned-maintainer risk without contributor infrastructure: Astrofox has 1.9k stars and is effectively dead since 2022. Wav2Bar-reborn maintainer explicitly warns of limited availability. Without a formal plugin API, community contributions require touching core code, which bottlenecks on maintainer review and causes projects to die when one person burns out.
- ISF license fragmentation: while ISF as a format is open, specific ISF scene files from Synesthesia's marketplace have per-scene commercial licensing restrictions. Importing community ISF scenes without checking their individual licenses can introduce undeclared commercial restrictions.
- WebGL2 as a ceiling: building on WebGL2 (butterchurn, modV) is safe today but forecloses compute shaders and modern GPU features. WebGPU is shipping in all major browsers as of 2024. Starting on WebGL2 now means a mandatory rewrite within 2-3 years or permanent capability gap vs WebGPU-native competitors.
- Fuzzy 'modular' branding without a real plugin ABI: many projects call themselves modular (modV, VSXu, ProkopHapala/MusicVisualizer) without shipping a documented, versioned, stable plugin interface. Without a concrete plugin ABI spec, 'modular' is marketing. Real modularity requires a defined interface that third-party code can compile against independently.
- Scope creep into video export vs real-time: Wav2Bar and Astrofox both prioritized video export, which creates fundamentally different technical requirements from real-time performance. A new project should pick one primary mode and excel at it before attempting both.
- Overlooking audio latency budget: projectM's SDL frontend has documented audio-visual sync lag. CAVA warns it is not intended for scientific use. A game-inspired visualizer must publish and enforce a latency budget (target: audio-to-visual under 16ms at 60fps), otherwise it will replicate the same complaints that haunt projectM in community forums.
